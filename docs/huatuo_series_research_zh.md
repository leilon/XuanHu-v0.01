# Huatuo 系列模型调研与对我们的训练启发

## 1. 先说结论

- `Agentic-RL` 放在最后是对的。
- `HuatuoGPT / Huatuo-II / HuatuoGPT-Vision / Huatuo-o1` 基本已经帮我们验证了三件事：
  - 中文医疗 SFT 数据是能把通用基模拉到较强医疗对话水平的。
  - 多模态医疗能力不能只靠把图片塞给模型，需要单独做视觉对齐和视觉指令训练。
  - 复杂医学推理和长链路决策，通常要在较强 SFT 基础上再做 reasoning / RL 类训练。
- 对我们这个项目来说，训练顺序建议改成：
  - `文本 SFT`
  - `多模态对齐 + 多模态 SFT`
  - `RAG 与长期记忆`
  - `QLoRA 周期增量适配`
  - `Agentic-RL / DPO / verifier-based optimization`

## 2. Huatuo 系列各自做了什么

### 2.1 HuatuoGPT

它解决的是“中文医疗对话基础能力”。

官方公开的信息显示，它的 SFT 数据主要由四类组成：

- ChatGPT 蒸馏的单轮医疗指令
- 真实医生单轮问答
- ChatGPT 蒸馏的多轮医疗对话
- 真实医生多轮对话

这一阶段最值得我们借鉴的不是模型结构，而是数据配方：

- 单轮和多轮要混合
- 真实医生风格和蒸馏后的高可读风格要混合
- 原始医疗咨询数据最好先清洗和重写，不要直接喂模型

### 2.2 Huatuo-II

它解决的是“如何把通用 LLM 更系统地适配成医疗模型”。

官方说明里给出的核心点有：

- 基于 Qwen2.5 系列做一阶段医学适配
- 使用了大规模医学预训练数据和约 `63.8` 万条医学指令数据
- 提出了面向医学任务的数据优先采样与分阶段适配思路

这给我们的启发是：

- 我们不需要复刻它的大规模医学继续预训练
- 但可以借它的数据思想重排我们的 SFT
- 具体来说，不要把所有医疗样本混成一个桶，而要按能力分层

建议把我们的文本 SFT 分成四层：

1. 医疗通识与问答层
2. 首程问诊与结构化病史采集层
3. 用药、安全边界、分诊层
4. 工具调用、RAG 引用、长期记忆融合层

### 2.3 HuatuoGPT-Vision

它解决的是“医学图文理解怎么接进医疗大模型”。

这条线对我们最重要，因为它已经证明：

- 医疗多模态不是只做 OCR
- 视觉基模上面仍然需要额外训练
- 单独的视觉对齐阶段很关键

官方训练说明里明确给了基于 `Qwen2.5-VL` 的流程，核心是三步：

1. 初始化视觉模型
   - 将 `Qwen2.5-VL` 结构与对应 LLM 权重对齐
2. 视觉对齐预训练
   - 使用 caption / 图文配对数据做视觉对齐
3. 视觉指令微调
   - 使用医疗 VQA、报告理解、图文问答数据做 SFT

所以对“Huatuo-Vision 有没有在 Qwen2.5-VL 上继续预训练或后训练”这个问题，答案是：

- 有
- 而且不是只做一层 SFT
- 至少包含一个视觉对齐阶段，再进入视觉指令微调阶段

### 2.4 Huatuo-o1

它解决的是“复杂医学推理”。

官方公开的信息显示，这条线更接近：

- 先构造高质量、可验证的复杂医学推理题
- 再做 reasoning-oriented SFT / optimization
- 最后再往更强的推理能力推进

对我们的启发是：

- `Agentic-RL` 不应该先做
- 应该先让模型具备稳定的医疗对话、结构化问诊、RAG 引用和工具调用能力
- 否则 RL 会放大错误行为，而不是稳定提高复杂决策能力

## 3. PubMedVision 对我们有什么用

`PubMedVision` 是 HuatuoGPT-Vision 公开出来的重要多模态数据资产之一。

它的价值主要有三点：

- 规模大，覆盖医学图像与图表问答
- 更接近学术医学图片，而不是普通自然图像
- 适合作为医学视觉对齐和医学 VQA 的底座数据

但它不能直接替代我们的真实目标场景，因为我们的项目更偏：

- 检验报告截图
- 化验单
- 门诊报告解读
- 患者上传的非标准手机拍照

所以我们的多模态 SFT 应该在 `PubMedVision` 基础上，再补：

- 检验单结构化解析数据
- 体检报告问答数据
- 胸片/影像的轻量问答数据
- 面向患者解释的通俗化输出数据

## 4. 对我们项目的 SFT 阶段应该怎么改

### 4.1 文本 SFT 不再只看“医疗问答”

建议拆成五个数据桶。

#### 数据桶 A：通用中文医疗问答

用途：

- 打牢医学常识、疾病解释、药品解释、患者友好表达

可用来源：

- `HuatuoGPT-sft-data-v1`
- `BillGPT/Chinese-medical-dialogue-data`
- `cMedQA-V2.0`
- 过滤后的 `shibing624/medical`

#### 数据桶 B：首程问诊轨迹

用途：

- 学医生如何完整问诊，而不是只会回答问题

这里必须补我们自己的数据。重点字段应该包括：

- 主诉
- 起病时间
- 症状演变
- 伴随症状
- 饮食与作息
- 流行病史 / 接触史 / 冶游史
- 既往史
- 过敏史
- 用药史
- 性别相关追问
- 初步检查建议
- 门诊 / 急诊 / 留观 / 收治判断

这一桶最适合用强 API 辅助合成，生成“首程问诊标准轨迹”。

#### 数据桶 C：安全边界与分诊

用途：

- 强化红旗症状识别
- 训练门诊可处理与需升级处理的边界

这桶数据要明确区分：

- 可门诊处理
- 建议尽快门诊
- 建议急诊
- 建议收治或进一步检查

#### 数据桶 D：RAG 与工具调用

用途：

- 让模型学会什么时候检索、什么时候引用、什么时候调工具

这里的数据不能只写“答案”，还要写：

- 检索意图
- 工具调用结果
- 最终引用依据

#### 数据桶 E：长期记忆与复诊

用途：

- 让模型学会读取历史病史，而不是每次从零开始

重点训练样本：

- 老用户二次复诊
- 有慢病背景的本次新症状
- 已有过敏史、长期用药史时的问药决策

### 4.2 多模态 SFT 建议拆成两段

不要直接把 `Qwen2.5-VL` 拿来做终版医疗问答 SFT。

建议先做：

1. 视觉对齐
   - 用 `PubMedVision`、医学 caption、图文配对数据
2. 医疗视觉指令微调
   - 用报告问答、医学 VQA、患者上传报告解释数据

这和 HuatuoGPT-Vision 的思路更一致，也更稳。

## 5. Agentic-RL 为什么放最后

因为它依赖前面的能力已经足够稳定。

如果以下能力还没成型：

- 首程问诊结构稳定
- 分诊标签可靠
- RAG 能给出依据
- 视觉输入能被正确理解
- 长期记忆不会乱写乱取

那么 RL 优化出来的往往只是“更会犯错”的策略。

所以我们项目更合理的顺序是：

1. `Qwen2.5-7B-Instruct` 文本医疗 SFT
2. `Qwen2.5-VL-7B-Instruct` 医疗视觉对齐与视觉 SFT
3. 接入 RAG、知识库与长期记忆
4. 用 QLoRA 做任务 adapter 和周期增量训练
5. 最后做 `DPO / ORPO / Agentic-RL`

## 6. 我们接下来最该补的东西

### 6.1 首程问诊数据

这是当前最关键的缺口。

Huatuo 系列帮我们证明了：

- 医疗问答可以训出来
- 多模态可以训出来
- 推理可以继续做

但它们并没有直接给我们“面向 C 端门诊首程问诊 + 长期记忆 + RAG + 多 Agent”的成套轨迹。

这部分需要我们自己补。

### 6.2 长期记忆数据

要补的不是“把历史对话全文塞回去”，而是：

- 结构化病史摘要
- 历史同类发作摘要
- 慢病与长期用药事实
- 需要过期的短期事实

### 6.3 Agentic-RL 数据

更适合作为最后一阶段：

- 从 benchmark / 首程病例包 / 真实使用日志出发
- 先有稳定的 `chosen` 轨迹
- 再构造带明确失败模式的 `rejected` 轨迹

## 7. 对我们的直接行动建议

### 现在应该继续做

- 用 `HuatuoGPT-sft-data-v1` 作为中文医疗 SFT 主料之一
- 把 SFT 重排成分桶训练，而不是混合训练
- 补首程问诊与复诊记忆数据
- 把视觉训练拆成“视觉对齐 + 视觉 SFT”

### 现在不应该急着做

- 一上来就做在线 Agentic-RL
- 把长期记忆直接写进参数
- 让 `Qwen2.5-VL` 同时承担全部文本与视觉主流程
