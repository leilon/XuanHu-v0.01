# Agentic-RL 合成数据方案说明

## 1. 为什么要自己合成 Agentic-RL 数据

医疗 Agent 的核心难点，不只是“回答得像不像医生”，而是：

- 会不会先追问关键病史
- 会不会在高风险场景及时升级
- 会不会调用正确工具
- 会不会基于证据给出 grounded 的建议

通用 RLHF 或公开 medical preference 数据，往往有几个问题：

- 不是多轮 Agent 任务，而是单轮问答偏好
- 很少覆盖工具调用错误
- 很少覆盖“漏问病史”“漏报红旗症状”这类真实失误
- 有些公开数据标签质量并不稳定

所以更合理的做法是：**用更强的 teacher/judge 模型，围绕真实医疗 Agent 任务，合成我们真正想优化的偏好对。**

## 2. 合成目标

我们希望最终得到适合 DPO / Preference Optimization 的样本：

- `prompt`
- `chosen`
- `rejected`
- `meta.case`
- `meta.failure_mode`
- `meta.judge`

其中：

- `chosen` 不是“完美答案”，而是更符合医疗 Agent 工作流的轨迹
- `rejected` 不是“胡说八道”，而是**看起来还像样、但关键环节犯错**的轨迹

这样训练出来的模型，才会真的学到：

- 先问什么
- 什么时候调工具
- 什么时候该保守
- 什么时候该升级就医

## 3. 整体流程

当前推荐流程分 4 步。

### 第一步：从种子问题生成隐藏病例包（case packet）

输入是一条真实医疗问题，例如：

- “我发烧咳嗽两天，需要吃什么药？”

teacher 模型先不直接回答，而是先把它转成一个“隐藏病例包”：

- 用户表面问题
- 隐藏病史
- 红旗风险
- 必须追问项
- 可能要调用的工具
- 可用证据片段
- 难度标签

一个简化版病例包示意：

```json
{
  "opening": "我发烧咳嗽两天，需要吃什么药？",
  "profile": {"age": "29", "sex": "female"},
  "hidden_facts": [
    "对布洛芬过敏",
    "已怀孕12周",
    "昨晚体温39.1度"
  ],
  "red_flags": ["pregnancy", "high_fever"],
  "must_ask": ["pregnancy_status", "drug_allergy", "temperature_range", "duration"],
  "likely_tools": ["rag_guideline", "drug_knowledge"],
  "difficulty": "hard"
}
```

这个步骤的价值是：后面生成正负样本时，teacher 和 judge 都围绕同一个“真实任务底稿”工作，而不是随便拍脑袋。

### 第二步：生成专家轨迹（chosen）

teacher 基于病例包生成一个更优的多轮 Agent 轨迹。

这个轨迹要包含：

- 是否先追问
- 追问什么
- 是否调用工具
- 调了什么工具
- 最终回答如何落地

一个典型的 `chosen` 轨迹应具备：

- 先问孕期、过敏史、体温
- 调药物知识库看禁忌
- 调 guideline 检索退烧建议
- 最终给出保守、安全、可执行的建议

### 第三步：生成有缺陷但合理的轨迹（rejected）

这里不是生成“明显错得离谱”的答案，而是生成**临床和产品里最常见的失败模式**。

当前我们重点合成 5 类失败模式：

- `premature_answer`
  - 还没问关键病史就直接给药
- `missed_red_flag`
  - 明明有高危信号却没建议就医
- `no_tool_use`
  - 明明应该查药物禁忌或指南，却直接凭经验回答
- `no_grounding`
  - 回答没有证据依据
- `missed_history`
  - 忘了问过敏、孕期、基础病、当前用药

这一步最重要的要求是：

- `rejected` 必须“看起来像正常回答”
- 但比 `chosen` 差在关键决策点上

这样训练才有意义。

### 第四步：judge 打分筛选

不是所有合成 pair 都要保留。

judge 要对 `chosen` 和 `rejected` 分别打分，维度包括：

- `safety`
- `coverage`
- `grounding`
- `tool_use`
- `overall`

只有当：

- `chosen` 明显优于 `rejected`
- 分差超过最小阈值（例如 `0.15`）

我们才保留这个样本。

这样可以过滤掉：

- 正负样本差异太小的 pair
- teacher/judge 自己生成失误的 pair
- 风格差异大但任务价值不高的 pair

## 4. 三个具体例子

### 例子一：发烧咳嗽 + 孕期 + 药物过敏

**种子问题**

- “我发烧咳嗽两天，需要吃什么药？”

**隐藏病例包**

- 用户怀孕 12 周
- 对布洛芬过敏
- 最高体温 39.1 度

**chosen**

- 先问是否怀孕、是否药物过敏、体温最高多少
- 调 `drug_knowledge`
- 调 `rag_guideline`
- 最终建议优先线下评估，避免自行使用布洛芬，并给出保守退热建议

**rejected（failure_mode=`premature_answer`）**

- 直接说“可以吃布洛芬或对乙酰氨基酚退烧”
- 没问孕期
- 没问过敏史

**为什么这个 pair 有价值**

- 它训练的是“先收集关键约束，再做推荐”，这是医疗 Agent 的核心能力。

### 例子二：胸痛 + 呼吸困难

**种子问题**

- “我今天下午突然胸口闷痛，还有点喘不上气。”

**隐藏病例包**

- 症状突发
- 呼吸困难
- 有高风险红旗

**chosen**

- 快速确认持续时间、疼痛性质、是否大汗/放射痛
- 立即给出急诊建议
- 不拖延到普通科普问答

**rejected（failure_mode=`missed_red_flag`）**

- 说“先休息观察一下，可能是焦虑或疲劳”
- 没建议急诊

**为什么这个 pair 有价值**

- 它训练的是“高风险优先级覆盖”，这比一般医学知识问答重要得多。

### 例子三：化验单白细胞升高

**种子问题**

- “我的血常规白细胞升高，帮我看看报告。”

**隐藏病例包**

- 有化验单
- 近两天乏力、低热
- 报告解释需要工具

**chosen**

- 调 `report_parser`
- 调 `rag_guideline`
- 说明白细胞升高可能提示炎症/感染，需要结合症状和其他指标综合判断
- 给出“何时应就医”的阈值

**rejected（failure_mode=`no_tool_use`）**

- 不看报告内容
- 直接说“问题不大，多喝水”

**为什么这个 pair 有价值**

- 它训练的是“遇到报告先工具解析，再回答”，而不是把多模态任务退化成普通聊天。

## 5. 这个方案和旧版脚本的区别

旧版脚本的问题是：

- `chosen` 直接来自当前原型 agent，本身能力太弱
- `rejected` 只是固定弱模板，区分度太粗糙
- 没有多轮追问
- 没有失败模式控制
- 没有 judge 过滤

新版方案的核心升级是：

- 用更强 teacher 先构建病例包
- 合成多轮轨迹而不是单句回答
- 明确控制失败模式
- 用 judge 做质量门控

所以新方案更适合训练：

- 多Agent 编排决策
- 工具调用习惯
- 安全升级能力
- grounded answer 习惯

## 6. 最适合 7B 的数据配比建议

如果我们最终用 `Qwen2.5-7B-Instruct` 做主文本模型，比较稳的配比是：

- 60%：高质量 SFT（Huatuo + 过滤后的 MedicalGPT + cMedQA 等）
- 25%：合成 Agentic-RL 偏好对
- 15%：人工筛过的公开 reward / preference 数据

原因是：

- 7B 先要学会“像个靠谱医疗助手一样说话”
- 再通过偏好优化学会“像个靠谱 Agent 一样决策”

如果一开始 RL 数据比例过大，7B 容易学到风格偏差，却没有足够扎实的医学基础表达能力。

## 7. 如何在项目里讲这件事

面试时可以这样概括：

“公开医疗偏好数据难以直接覆盖多轮问诊、工具调用和安全升级这类 Agent 决策场景，所以我设计了一套基于强 teacher 的合成 pipeline。它先把真实用户问题转成隐藏病例包，再分别生成专家轨迹和特定失败模式的负轨迹，最后通过 judge 对安全性、信息覆盖度、工具使用和 grounding 进行打分，只保留高质量偏好对。这样得到的 RL 数据更贴近我们真实想优化的医疗 Agent 行为。”
